**主要ツール:**

## 2. AIペアコーディング

### 2.1 従来のペアプロ vs AI

| 項目       | 人間ペア     | AIペア       |
| ---------- | ------------ | ------------ |
| 稼働時間   | 勤務時間のみ | 24時間       |
| 疲労       | あり         | なし         |
| 質問の遠慮 | あり         | 不要         |
| コスト     | 人件費×2     | サブスク料金 |

### 2.2 基本ワークフロー

```
1. 要件を言葉で説明
   ↓
2. AIが設計案を提示
   ↓
3. 対話しながら調整
   ↓
4. 実装
   ↓
5. AIにレビュー依頼
   ↓
6. 改善・リファクタリング
```

### 2.3 効果的なプロンプトの書き方

**❌ 悪い例:**

```
「ログイン画面を作って」
```

**✅ 良い例:**

```
「React + TypeScript で、Firebase Authentication を使った
 ログイン画面を作成してください。

 要件:
 - メールアドレスとパスワードでログイン
 - バリデーション: メールアドレス形式、パスワード6文字以上
 - エラー表示: 日本語
 - スタイリング: Tailwind CSS
 - ファイル名: LoginForm.tsx」
```

**ポイント:**

1. 具体的な要件
2. 技術的制約
3. 期待する出力形式

### 2.4 注意点

- ✅ AIの出力を必ず読む
- ✅ 意図を理解してから採用
- ❌ コピペして動けばOKは危険
- ❌ セキュリティ部分を無検証で採用

## 3. SDD（Specification-Driven Development）

### 3.1 SDDとは

**仕様書を先に書いてから実装する開発手法**

従来: コードを書きながら考える  
SDD: 仕様を明確にしてから、AIに実装させる

### 3.2 なぜAI時代にSDDなのか

- AIは「何を作るか」が明確なら高品質な実装を高速生成
- 曖昧な指示では期待と異なる出力
- **仕様の明確化 = AIを活用する鍵**

### 3.3 SDDの進め方

```
1. 機能要件を箇条書き
   ↓
2. Claudeに仕様書のドラフトを作成させる
   ↓
3. レビュー・修正
   ↓
4. 仕様書をもとにClaudeCodeで実装
   ↓
5. テスト・検証
```

### 3.4 仕様書の例

```markdown
# ユーザー管理画面 仕様書

## 1. 概要

Firestoreのユーザー情報を一覧表示し、検索・編集・削除を行う

## 2. 画面仕様

- ヘッダー: 「ユーザー管理」
- 検索バー: 名前・メールで検索
- テーブル: 名前、メール、権限、登録日、操作
- ページネーション: 20件/ページ

## 3. 権限制御

- admin: 全操作可能
- user: 閲覧のみ
```

### 3.5 SDDのメリット

1. チーム内での認識統一
2. 後から見ても何を作ったかわかる
3. AIへの指示が明確
4. 品質向上（エッジケースに気づく）
5. レビューが効率的

## 4. デモ: 今日の気持ちアプリ

### 4.1 アプリ概要

**機能:**

- 気持ちを記録
- 一言を記録
- 過去の記録を一覧表示
- LocalStorageに保存

**技術スタック:**

- React + TypeScript (Vite)
- Tailwind CSS
- LocalStorage

### 4.2 デモの流れ

**Step 1: Claudeで仕様書作成（5分）**

```
 Next.js (Pages Router) + MUI + TypeScript で「今日の気分」記録アプリを作成してください。
  既存のNext.jsプロジェクトがあるので、そこに実装してください。

  ## 技術スタック
  - Next.js (Pages Router) ※既にセットアップ済み
  - MUI (@mui/material, @mui/icons-material)
  - recharts（グラフ描画）
  - canvas-confetti（紙吹雪演出）
  - データ保存: localStorage
  - ページ遷移なし（1ページ完結、状態で画面切替）

  ## 画面構成

  ### ログイン画面
  - メールアドレス + パスワード入力（認証は仮実装、入力があればOK）
  - ゲストログインボタン（認証なし、即ログイン）
  - グラデーション背景 + カード型UIのおしゃれなデザイン
  - ログイン状態はlocalStorageで保持

  ### メイン画面（ログイン後）
  - ヘッダー: ユーザーアバター、アプリ名、ログアウトボタン（sticky）

  #### 今日の気分記録カード
  - 3つの気分から選択: ☔雨(1) / ☁️ 曇り(2) / ☀️ 晴れ(3)
  - 気分ボタンは選択時にボーダー+背景色でハイライト、ホバーで拡大
  - 一言メモ入力欄（複数行）
  - 記録済みなら「記録済みです」表示 + 編集ボタン

  #### 過去の気分追加ボタン
  - 日付を指定して過去の記録を追加可能

  #### タブ切り替え（カレンダー / 統計 / 履歴）

  **カレンダータブ:**
  - 月間カレンダーに気分の絵文字を表示
  - 月の前後移動対応、今日をハイライト
  - 日付クリックで追加/編集ダイアログを開く
  - 未来の日付はクリック不可（薄く表示）

  **統計タブ:**
  - 7日 / 30日の期間切り替え（Chip）
  - 各気分の割合をプログレスバー（%表示）で可視化
  - rechartsの折れ線グラフで気分の推移を表示（Y軸は絵文字）

  **履歴タブ:**
  - 記録を日付降順でカード一覧表示
  - 各カードに気分絵文字、日付、気分ラベルChip、メモ
  - 各カードに編集ボタン（鉛筆アイコン）

  ## 追加機能

  ### 編集ダイアログ
  - 日付、気分選択、メモを変更可能
  - 新規追加と編集で共通のダイアログを使用
  - 日付は未来を選択不可に制限

  ### 気分フィードバック
  - 記録時に気分に応じたランダムな一言メッセージをSnackbarで表示
    - 雨: 「無理しないでくださいね」等
    - 曇り: 「小さな幸せを見つけてみて」等
    - 晴れ: 「素敵な日ですね！」等

  ### 紙吹雪アニメーション
  - 記録時にcanvas-confettiで演出
  - 晴れ: 暖色で派手（100粒）、曇り: グレー系（40粒）、雨: 青系+重力強め（30粒）

  ### 全データ削除
  - ページ下部に「全データを削除」ボタン（赤）
  - 確認ダイアログ付き（警告メッセージ表示）
  - localStorageのデータを完全削除

  ## デザイン方針
  - MUIのカスタムテーマ: primary=#5C6BC0, secondary=#FF8A65, borderRadius=12
  - 背景はグラデーション、カードにシャドウ
  - Fade アニメーションで画面切替
  - 全体的に丸みのある、温かみのあるデザイン

  ## データ構造
  - ユーザーごとに `mood_records_{user}` キーでlocalStorageに保存
  - MoodRecord: { date: string (YYYY-MM-DD), mood: 1|2|3, memo: string }

  実装はpages/index.tsxの1ファイルに集約してください。


```
